# QGI Paper Series — authoring and release skill

## Scope

One canonical paper series, many SKUs. The series covers:

- **Preprints** in `papers/preprints/{A..F}-*/paper.md` — each a standalone, citable preprint.
- **Executive whitepaper** in `papers/whitepaper-executive/whitepaper.md` — one math-lite CIO/procurement version.
- **Blog derivatives** in `papers/blog/{A..F}-*/{venue}.md` — six venues per paper.

Paper A is the canonical reference; every other paper cites it. The evaluation paper (paper G) is held back until the evaluation-agreement release clears numbers for publication.

## Author block (apply to every piece)

```yaml
author:
  - name: Sam Sammane
    email: sam@qgi.dev
    affiliation: Chief Technology Officer and Founder, Quantum General Intelligence, Inc.
    orcid: ""
```

In body text prefer "Dr. Sam Sammane" the first time the author is named in each document, and "Sammane" on subsequent prose references. Byline on blog derivatives is "Dr. Sam Sammane — CTO and Founder, Quantum General Intelligence".

## Shared typesetting conventions

1. Every preprint and the whitepaper include `papers/common/_metadata.yaml` via pandoc's `--metadata-file` flag.
2. The shared DOCX style is `papers/common/reference.docx`, generated by `papers/common/build_reference_docx.py`.
3. The shared figure corpus is `papers/common/figures/`. Preprints reference it via `../../common/figures/`.
4. The shared colour palette is:
   - `qgiblue` `#2E4A8F` (headings)
   - `qgiblueSoft` `#4A68B0` (subheadings)
   - `qgigray` `#4B5561` (meta / captions)
   - `qgimuted` `#6B7280` (date / page numbers)
   - `qgirule` `#C7D2E4` (horizontal rules)
5. Fonts: Latin Modern for PDF, Calibri for DOCX body, Consolas for monospace.
6. Inline math MUST use `$...$` dollars, not `\(...\)`, because Pandoc 3.1.x does not reliably route `\(...\)` through math mode in this corpus.
7. Section cross-references use the form `[§8](#anchor)` — plain pandoc, no `pandoc-crossref`.

## Preprint hygiene checklist (every preprint must have)

- Preprint banner — `Preprint — v1.0, YYYY-MM-DD. Not peer-reviewed. Comments: research@qgi.dev`
- Abstract (≤ 250 words)
- Keywords (≥ 8)
- Related work / prior art section (for review-style preprints this is the bulk)
- Limitations section
- Competing interests — Sam is CTO/Founder of QGI
- Data availability — benchmark corpora are proprietary, released under evaluation agreement
- Ethics note — dual-use disclosure where relevant
- Version history block at the end
- "Cite this as" block with BibTeX stub pointing at the Zenodo DOI placeholder
- References (≥ 10 for method notes, ≥ 30 for review-style papers)

## Release cadence

Do not release all six preprints on the same day. The recommended cadence is:

| Week | Paper | Why this order |
|---|---|---|
| 1 | A (canonical) | Anchors every other citation |
| 3 | F (beyond-RAG review) | Positions QGI as thinker; reframes A in a landscape |
| 5 | C (agent memory) | New audience (agent frameworks) |
| 7 | B (Q-Prime embedding) | Depth follow-up |
| 9 | D + E (method notes) | Smaller, paired release |
| when cleared | G (evaluation paper) | Numbers release |

## Publishing venues per release

1. **arXiv** — cs.CL primary; cs.AI, cs.IR cross-list. Canonical preprint URL.
2. **Zenodo** — DOI, mirrors the PDF.
3. **HuggingFace Papers** — auto-indexed from arXiv.
4. **Company blog** — canonical long-form (`qgi.dev/blog/...`).
5. **LinkedIn Pulse** — executive audience.
6. **Substack** — researcher audience.
7. **Medium / Towards Data Science** — developer audience.
8. **HackerNews** — short launch post + link.
9. **Twitter/X** — one 10–12-tweet thread.

Each secondary venue uses a distinct headline and lead angle — no duplicate bodies. Every secondary page uses `<link rel="canonical">` pointing at arXiv or the paper PDF landing page. Schema.org `ScholarlyArticle` JSON-LD on the company-blog canonical.

## Build recipe

From repository root:

```bash
make -C papers all          # render every preprint and whitepaper
make -C papers preprints    # preprints only
make -C papers whitepaper   # executive whitepaper only
make -C papers figures      # regenerate rasterized figures (if needed)
make -C papers reference    # regenerate papers/common/reference.docx
make -C papers clean        # remove all generated PDFs/DOCXs
```

Required toolchain:

- `pandoc` ≥ 3.1
- `xelatex` (TeX Live)
- A Python venv with `python-docx`, `pypdf`, `cairosvg` (see `papers/Makefile` for the venv path used by CI)

## Naming conventions

- Preprint folder: `papers/preprints/<letter>-<kebab-title>/`
- Source: `paper.md`
- Outputs: `paper.pdf`, `paper.docx`
- Blog folder: `papers/blog/<letter>-<kebab-title>/`
- Blog files: `company-blog.md`, `linkedin.md`, `substack.md`, `medium.md`, `hackernews.md`, `twitter.md`
- Executive whitepaper: `papers/whitepaper-executive/whitepaper.{md,pdf,docx}`

## When adding a new preprint

1. Duplicate the closest existing preprint folder.
2. Replace the YAML title/abstract/keywords; keep the shared header-includes block intact.
3. Import figures you need from `../../common/figures/`.
4. Cite paper A at the first appropriate section.
5. Append an entry in `papers/README.md` manifest.
6. Add a Makefile target.
7. Run the preprint hygiene checklist.

## When adding a new blog venue

1. Add a new file `<venue>.md` under `papers/blog/<paper>/`.
2. Update the "Blog derivatives" section of `papers/README.md`.
3. Keep the word-count target aligned with venue norms:
   - company-blog: 1,200–2,000 words
   - linkedin: 600–1,000 words
   - substack: 1,000–1,600 words
   - medium: 900–1,300 words
   - hackernews: 150–300 words
   - twitter: 10–14 tweets, each ≤ 280 chars

## Do-nots

- Do not salami-slice a single contribution across multiple preprints.
- Do not duplicate the preprint PDF body verbatim on a blog — use a different hook per venue.
- Do not publish numerical benchmark results in any preprint other than paper G.
- Do not push `papers/` changes to the public `Quantum-General-Intelligence/QGI-Embedding-Model-Q-Prime` remote; the series is intended for a private staging repo.
